# Provenance Guard

A backend system that any creative-sharing platform could plug into to estimate whether submitted text was written by a human or generated by AI. Rather than forcing a binary verdict, Provenance Guard combines two independent detection signals into a calibrated **confidence score**, surfaces a plain-language **transparency label**, records every decision in a structured **audit log**, and lets creators **appeal** classifications they believe are wrong.

The goal is not perfect AI detection — that is an unsolved problem — but an honest system that communicates uncertainty and gives creators recourse.

> **Build status:** Milestones 1–4 complete (submission endpoint, both detection signals, confidence scoring, audit log). Milestone 5 features (transparency-label endpoint wiring, appeals, rate limiting) are noted as _Pending M5_ below.

```bash

source .venv/bin/activate
pip install -r requirements.txt

# Create a .env (gitignored) with your Groq key:
echo 'GROQ_API_KEY=your_key_here' > .env

```

The server listens on port **5000** by default. On macOS, port 5000 is held by the AirPlay Receiver (it returns HTTP 403), so run with an alternate port:

PORT=5001 python app.py

## Architecture Overview

A submission flows through the system as follows:

1. **`POST /submit`** receives JSON `{ text, creator_id }` and validates both fields.
2. **Signal 1 (Groq LLM)** judges the text holistically and returns an AI-likelihood score in `[0, 1]`.
3. **Signal 2 (stylometric heuristics)** independently measures structural properties of the text and returns its own `[0, 1]` score.
4. **Confidence scoring** combines the two signals with a weighted average into a single `confidence` score and maps it to an attribution (`likely_human` / `uncertain` / `likely_ai`).
5. **Transparency label** text is selected from the attribution band (_wired in M5_).
6. **Audit log** records the decision — both signal scores, the combined confidence, attribution, and timestamp — as one JSON line.
7. The structured JSON response is returned to the caller, including a unique `content_id` used later for appeals.

The full diagram for both the submission and appeal flows lives in [`planning.md`](planning.md#architecture).

## Detection Signals

Provenance Guard uses two **genuinely independent** signals — one semantic, one structural — so the combination is more informative than either alone.

### Signal 1 — Groq LLM (semantic)

Sends the text to Groq's `llama-3.3-70b-versatile` and asks it to estimate the probability the passage was AI-generated, considering tone, coherence, repetition, and phrasing. Returns a float in `[0, 1]` (1 = strongly AI).

- **What it captures:** holistic semantic and stylistic coherence — the "feel" of AI text (formulaic transitions, generic hedging, absence of personal voice) that is hard to express as fixed rules.
- **What it misses:** it can flag polished, professional, or carefully edited human writing as AI, because those styles overlap with AI output. It is also non-deterministic in principle and depends on an external API.

### Signal 2 — Stylometric heuristics (structural)

Computes three measurable properties of the text in pure Python and maps each to a 0–1 AI-likeness sub-score:

| Metric | What it measures | Direction | Weight |
|--------|------------------|-----------|-------:|
| Sentence-length variation (coefficient of variation) | Burstiness — how much sentence length varies | Low variation → AI-like | 0.55 |
| Type-token ratio (first 100 words) | Vocabulary diversity | Low diversity → AI-like | 0.20 |
| Structural punctuation density (`, ; : — ( ) "` per word) | Use of formal/subordinate punctuation | High density → AI-like | 0.25 |

- **What it captures:** structural uniformity. AI text tends to be even and polished; human writing is more variable ("bursty"). This is completely independent of the LLM's semantic judgment.
- **What it misses:** it scores formal, academic, legal, and technical human writing as AI-like, because those styles are uniform and punctuation-heavy. The type-token ratio is also unreliable on short texts (which is why it is weighted least and windowed).

## Confidence Scoring

Both signals produce a score in `[0, 1]`. They are combined with a fixed weighted average:

```
confidence = 0.65 * llm_score + 0.35 * heuristic_score
```

The LLM is weighted more heavily because it captures semantic nuance the heuristics cannot; the heuristic still contributes an independent structural check. The combined score maps to three attribution bands:

| Combined score | Attribution |
|---------------:|-------------|
| 0.70 – 1.00 | `likely_ai` |
| 0.31 – 0.69 | `uncertain` |
| 0.00 – 0.30 | `likely_human` |

The `uncertain` band is intentionally wide. Because a false positive (labeling a human's work as AI) is worse than a false negative on a writing platform, the system reserves a broad middle region and only commits to `likely_ai` when evidence is strong.

### How I validated that scores are meaningful

I tested the pipeline on inputs chosen to span the confidence range — clearly AI, clearly human, and two borderline cases — and printed both signal scores separately to confirm the combination behaved sensibly. The scores spread cleanly from 0.11 to 0.83, and the two borderline cases correctly landed in `uncertain` rather than flipping to a confident verdict.

### Example submissions (actual scores)

| Input | `llm_score` | `heuristic_score` | `confidence` | Result | Confidence level |
|-------|------------:|------------------:|-------------:|--------|------------------|
| AI-generated listicle ("Firstly… Secondly… In conclusion…") | 0.93 | 0.37 | **0.7355** | `likely_ai` | High |
| Casual human review ("ok so i finally tried that ramen place…") | 0.12 | 0.15 | **0.1299** | `likely_human` | High |
| Lightly edited AI ("I've been thinking about remote work…") | 0.42 | 0.47 | **0.4388** | `uncertain` | Low |

The first two are **high-confidence** cases far from the midpoint and produce noticeably different scores (0.74 vs 0.13). The third sits near the middle — a **low-confidence** case the system honestly refuses to commit on.

A notable validation result: formal human writing about monetary policy scored `llm=0.83, heuristic=0.41, confidence=0.682` — the LLM and the uniformity heuristic both leaned AI, but the wide `uncertain` band kept it from a false `likely_ai` verdict. This is the false-positive protection working as designed.

## Transparency Label

The label shown to readers is plain-language and avoids jargon ("classifier," "probability," "score"). Its text changes with the attribution band. The three variants:

| Attribution | Label Text |
|-------------|------------|
| **High-confidence AI** (`likely_ai`) | "This content appears likely to be AI-generated. We are showing this label to provide additional context. The creator may appeal this decision." |
| **Uncertain** (`uncertain`) | "We are not confident enough to determine whether this content was written by a human or generated by AI. No final attribution claim is being made." |
| **High-confidence human** (`likely_human`) | "This content appears likely to be written by a human. No AI-generation label is being shown." |

## Audit Log

Every decision is written as one JSON line to `logs/audit.jsonl` (structured JSON Lines, not console output). `GET /log` returns recent entries newest-first. A submission entry:

```json
{
  "event_type": "submission",
  "content_id": "c4ec5110-bb9f-41e0-bf3f-ad7e57d8a905",
  "creator_id": "demo-ai",
  "timestamp": "2026-06-30T22:08:26.366822+00:00",
  "llm_score": 0.93,
  "heuristic_score": 0.3743,
  "confidence": 0.7355,
  "attribution": "likely_ai",
  "status": "classified"
}
```

Each entry includes the attribution result, confidence score, both individual signal scores, and a timestamp. _Pending M5: appeal entries will appear in the log alongside the original classification._

## Known Limitations

**Formal, academic, or technical human writing is the system's hardest case.** Such writing has uniform sentence structure and dense, formal punctuation, so the stylometric signal's burstiness and punctuation-density metrics both push it toward "AI-like." The LLM signal often agrees, because polished formal prose reads similarly to AI output. In testing, a genuine human paragraph about monetary policy reached `confidence = 0.682` — only the deliberately wide `uncertain` band kept it from a false `likely_ai` verdict. A creator who writes in a formal register is therefore the most likely to be misclassified, which is exactly why the appeals workflow exists.

**Very short text** (poems, captions, headlines) also degrades both signals: stylometric metrics need enough sentences to be meaningful, and the type-token ratio in particular is inert on short input. The system should — and generally does — fall back to `uncertain` when evidence is thin.

## Spec Reflection

**How the spec helped:** Writing the exact scoring weights (`0.65 / 0.35`) and thresholds (`0.30 / 0.70`) in planning.md *before* coding made the scoring logic unambiguous to implement and gave me concrete targets to test against. When I tested, I was checking against numbers I had already committed to, not inventing them after the fact.

**How the implementation diverged:** My plan assumed the LLM signal would naturally span the full 0–1 range. In practice the raw model returned conservative, coarse round numbers (capping around 0.8), which made the `likely_ai` band effectively unreachable — even a blatant AI listicle landed at `confidence = 0.68`. Rather than lower the 0.70 threshold I had specified, I kept the threshold and instead rewrote the LLM prompt to require two-decimal precision and to use the full range, reserving ≥0.85 for clear AI hallmarks. After that change, clearly-AI text crossed into `likely_ai` while the conservative-by-default behavior was preserved.

The generated scoring logic matched my spec, but testing surfaced a real problem: nothing ever reached `likely_ai` because the LLM emitted conservative round numbers. I overrode the easy fix (lowering the threshold) and instead had the prompt rewritten to use the full numeric range — a decision that kept my documented thresholds intact while fixing reachability. I verified the fix by re-running seven labeled test inputs and confirming all three attribution bands were reachable.
